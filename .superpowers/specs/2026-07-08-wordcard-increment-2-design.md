# FLQ-18 — WordCard (Increment 2): design spec

- Дата: 2026-07-08
- Задача: FLQ-18 — WordCard follow-ups: переводы-мультиполя, contextual AI, состояния AI,
  layout'ы, cleanups (Increment 2 после FLQ-5)
- Референсы: `docs/superpowers/specs/FLQ-5-word-card-design.md` (Increment 1),
  ADR-0003 (AI), ADR-0005 (статусы/confidence), `backlog/tasks/flq-18 - *.md`
- Источники требований: финальное ревью ветки FLQ-5 + runtime-верификация 2026-07-08
  (playwright, живой стек)

## 1. Scope

**Входит:**
- Переводы как список полей-вариантов: сейв по Enter/blur, `+`/`✕` на hover,
  «первое поле = primary». Debounce-автосейв перевода удаляется.
- Backend: `PATCH`/`DELETE` для переводов, серверный расчёт `is_primary`,
  уникальность текста варианта, data-миграция для существующих дублей.
- Contextual AI: `context_text` = предложение, содержащее слово (FLQ-5 §3.2).
- Состояния AI: info-note при выключенном AI (503), inline-ошибка + retry при
  реальных сбоях, AI-подсказка также для `known`.
- Layout'ы: collapsed top-2 подсказки, expand-persist в readerStore,
  отдельный ignored-layout.
- Cleanups и тесты из backlog-задачи FLQ-18.

**НЕ входит:** phrase-карточка, внешние словари-ссылки, TTS, AI-грамматика,
markdown в заметках, явный выбор primary (звёздочка), reorder вариантов.

## 2. Переводы: мультиполя (ревизия FLQ-5 §4.4)

### 2.1 UI

Блок «Перевод» — вертикальный список полей ввода, по одному на сохранённый
вариант, в порядке создания (порядок из `lookup.translations.all`). Пустое поле
не висит под списком постоянно: новое поле появляется только по `+`; у слова
без единого перевода показано одно пустое поле.

Поведение:
- **Enter** в поле — сохранить (основной жест). **Blur** — тоже сохранить
  (страховка от потери ввода при закрытии карточки хоткеем/кликом мимо).
- Поле с несохранённым текстом → `POST` (новый вариант); отредактированный
  текст существующего варианта → `PATCH`; текст не изменился → no-op.
- **Hover на сохранённом поле** показывает справа внутри поля две иконки:
  `+` — добавить пустое поле (фокус в него); поле появляется в конце списка —
  порядок вариантов равен порядку создания, поэтому «после текущего» поле всё
  равно оказалось бы в конце после сохранения (ревизия по итогам финального ревью);
  `✕` — удалить вариант (`DELETE`).
- Очистить поле существующего варианта и Enter/blur → тоже удаление варианта
  (симметрично `✕`; пустых строк в БД не бывает).
- Удалены все варианты → остаётся одно пустое поле ввода.
- Ввод текста, совпадающего с существующим вариантом, в новом поле + Enter →
  backend возвращает существующую строку (§2.4), дубль не появляется, лишнее
  поле схлопывается при обновлении списка.
- На `new`-слове Enter в поле перевода сначала авто-создаёт item
  `tracked/0` (правило FLQ-5 §2.1 не меняется), затем `POST` перевода.
- Ошибка сохранения: inline-текст «Не удалось сохранить» под блоком +
  повторный Enter/blur ретраит; ввод не очищается (как в Increment 1).

Debounce-таймер перевода (800ms) удаляется полностью — вместе с ним исчезает
источник мусорных строк. Заметка **остаётся** на сейве по blur (textarea,
Enter там — перенос строки); это фиксируется как осознанная ревизия §4.4.

### 2.2 Primary

- Верхнее поле (самый ранний вариант) — primary. Никакого UI для выбора.
- `is_primary` рассчитывает **backend**: `POST` ставит `true` только если у
  `(owner, item, target)` ещё нет переводов, иначе `false`; поле `is_primary`
  из тела запроса игнорируется (из схемы удаляется).
- `DELETE` primary-варианта промоутит самый ранний оставшийся
  (`created_at asc`) того же target.
- `PATCH` меняет только `translation_text` (и `source_type` → `user`,
  если правился текст не-user варианта).

### 2.3 Suggestions (ревизия FLQ-5 §4.2)

- Блок подсказок = только AI (`✦`) и Wiktionary (`📘`). Собственные переводы
  из подсказок убираются (они теперь поля выше). Заголовок — «Подсказки».
- Клик `+` на подсказке добавляет её вариантом в конец списка полей
  (`POST`, `source_type` = `ai`/`dictionary`). Если вариант с таким текстом
  уже есть — backend возвращает существующую строку, дубль не создаётся,
  UI ничего не добавляет.
- aria-label кнопки включает источник: `Добавить перевод (📘): строитель` —
  устраняет дубли aria-label при одинаковом тексте из разных источников.

### 2.4 Backend: эндпоинты и модель

| Метод | Путь | Назначение |
|---|---|---|
| `POST` | `/api/vocabulary/items/{kind}/{id}/translations` | добавить вариант; идемпотентно по тексту |
| `PATCH` | `/api/vocabulary/items/{kind}/{id}/translations/{translation_id}` | изменить текст варианта |
| `DELETE` | `/api/vocabulary/items/{kind}/{id}/translations/{translation_id}` | удалить вариант (+перенос primary) |

- `POST`: тело `{target_language_code, translation_text, source_type}` —
  без `is_primary` (см. §2.2). Существующий текст → 200 с существующей строкой.
- `PATCH`: тело `{translation_text}`. Если новый текст совпадает с другим
  вариантом того же target → 409 (UI показывает «Такой вариант уже есть»
  и оставляет ввод).
- `DELETE`: 200 `{translations: [...]}` (обновлённый список — UI перерисовывает
  поля без повторного lookup).
- Владелец проверяется как в остальных ручках (`_owned_item`); строка перевода
  должна принадлежать `(owner, item)`.
- Сервис: `update_translation`, `delete_translation` в
  `flinq/modules/vocabulary/service.py`; `add_translation` теряет параметр
  `is_primary` и получает дедуп по тексту.

### 2.5 Миграция `0008_translation_variants`

1. Data-фикс: удалить точные дубли
   `(owner_user_id, item_kind, item_id, target_language_code, translation_text)`,
   оставляя строку с `is_primary = true`, при равенстве — самую раннюю.
2. Уникальный индекс `uq_personal_translations_text` по этой пятёрке.
3. Существующий partial-индекс единственности primary не меняется.

Строки-«огрызки» набора («мож», «мощн») **не** вычищаются автоматически —
отличить их от осмысленных вариантов нельзя; теперь они видимы как поля
и удаляются `✕` вручную (или в FLQ-6).

## 3. AI: контекст и состояния

### 3.1 Contextual AI (закрывает FLQ-5 §3.2)

- ReaderPage вычисляет текст предложения, содержащего выбранный токен (по
  границам sentence-сегментов уже загруженного контента; работает и в page-,
  и в sentence-режиме), и передаёт в WordCard проп `sentenceText: string | null`.
- WordCard шлёт `context_text = sentenceText ?? word.t` (фолбэк — слово, как
  сейчас). `surface_text`, `lesson_id` — без изменений.
- `queryKey` AI-запроса включает контекст, чтобы карточка одного слова в разных
  предложениях не отдавала чужой кэш.

### 3.2 Состояния AI-блока (закрывает FLQ-5 §5)

- HTTP **503** от `POST /api/ai/translate` (kill-switch/выключен) → серый
  info-note «AI-переводы отключены» (стиль muted, не destructive), без retry.
- Прочие ошибки (сеть, 5xx, timeout) → inline-ошибка «Не удалось получить
  AI-перевод» + кнопка «Повторить» (refetch). Остальная карточка работает.
- API-клиент должен пробрасывать HTTP-статус ошибки (проверить `client.ts`;
  если статус недоступен — добавить его в тип ошибки клиента).
- `wantAi` расширяется: AI-подсказка запрашивается для `new` **и** `known`
  (FLQ-5 §4.3); для `tracked`/`ignored` — нет.

## 4. Layout'ы

### 4.1 Collapsed / Expanded (ревизия FLQ-5 §4.2)

- **Collapsed:** header (слово + ✕), поля переводов, **top-2 подсказки**
  (AI первыми, затем словарь), footer `🗑 [1-4] ✓`, шеврон.
- **Expanded:** + полный список подсказок, теги (chip-input), заметки.
- «Тег +» в collapsed из FLQ-5 §4.2 убирается осознанно (ревизия): теги —
  только в expanded, как реализовано в Increment 1.
- Expand-состояние переезжает из локального `useState` в `readerStore`
  (Zustand, in-memory, глобальный флаг `wordCardExpanded`) — переживает
  переоткрытие карточки, сбрасывается перезагрузкой страницы.

### 4.2 Ignored-layout (закрывает FLQ-5 §4.3)

Для `status = 'ignored'` тело карточки заменяется: строка «Игнорируется» +
подсказка «Выберите уровень 1–4 или ✓, чтобы вернуть слово в изучение».
Поля переводов, подсказки, теги/заметки и шеврон скрыты. Footer остаётся и
работает как реактивация (клик `1..4` → tracked/N, `✓` → known — уже работает).

## 5. Cleanups (из backlog FLQ-18)

- Общий хелпер `ensureItem`-then-mutate вместо 4 повторов в WordCard.tsx.
- `lookup()` реюзает `_list_tags` вместо inline-запроса тегов (service.py).
- Убрать неиспользуемый `item_id` из `_resolve(kind, item_id)`
  (backend/src/flinq/api/vocabulary.py).
- Убрать каст `'token' as ItemKind` в useWordCard.ts.
- Регрессионный тест ReaderPage: выбор слова рендерит настоящий WordCard.

## 6. Тестирование

- **Backend:** POST-дедуп по тексту (возврат существующей строки); серверный
  `is_primary` (первый вариант true, последующие false); PATCH текст /
  409 на дубль; DELETE обычного и primary-варианта (перенос primary на самый
  ранний); миграция 0008 на данных с дублями; владение строкой перевода.
- **Frontend (WordCard.test.tsx + новые):** Enter создаёт вариант; Enter на
  отредактированном поле патчит; пустое поле + Enter удаляет; `✕` удаляет;
  `+` добавляет пустое поле; последний удалённый вариант оставляет пустое
  поле; 503 → info-note без retry; сетевая ошибка → ошибка + «Повторить»;
  подсказки без user-переводов, top-2 в collapsed; ignored-layout; персист
  expand через readerStore; `context_text` = переданное предложение.
- **Integration (ручная проверка):** клик по new-слову → Enter-сейв перевода →
  токен голубой (tracked/0) → уровень 2 → жёлтый; `+` добавляет второй
  вариант; `✕` верхнего переносит primary.

## 7. Открытые вопросы (не блокируют)

- Явный выбор primary (звёздочка) и reorder — при необходимости в FLQ-6+.
- Автоочистка «огрызков» набора из старых данных — вручную через `✕`/FLQ-6.
