# ADR-0003 — LLM provider: OpenAI-compatible adapter

- Статус: Accepted
- Дата: 2026-04-11
- Связан со: `docs/specs/2026-04-11-mvp-product-alignment-design.md` §5, §10.2

## Контекст

Спецификация §7.7 требует, чтобы AI-слой был проектирован как оркестратор, с поддержкой облачных и локальных LLM, с возможностью routing'а, fallback'а и полного отключения внешнего AI. §16.3 дополнительно требует, чтобы базовый режим продукта работал без AI-провайдера вообще.

В MVP AI используется ровно в одном сценарии — контекстный перевод выделенного слова или фразы (§5 основного документа). Этот сценарий простой: короткий prompt, короткий ответ, одна модель, одна пара языков за запрос.

Перед нами выбор стратегии интеграции с LLM-провайдерами:

- **A.** Один общий адаптер, совместимый с OpenAI Chat Completions API, и считать все поддерживаемые бэкенды «реализациями этого интерфейса». OpenAI, OpenRouter, vLLM, LM Studio, LocalAI, Ollama (через её OpenAI-compat endpoint) — все поддерживают этот формат.
- **B.** Отдельные адаптеры для каждого провайдера (Anthropic, Google, OpenAI, Ollama) с общим абстрактным интерфейсом.
- **C.** Использовать абстракцию вроде LangChain / LiteLLM / Pydantic AI как сторонний слой.

## Решение

Принят **вариант A**: единственный адаптер, совместимый с OpenAI Chat Completions API. Конфигурация адаптера задаётся тремя параметрами:

- `base_url` — endpoint;
- `api_key` — токен (опционально, некоторые локальные бэкенды не требуют);
- `model` — имя модели.

Всё, что не укладывается в этот интерфейс, в MVP не поддерживается.

**Почему именно OpenAI-compatible, а не, например, LiteLLM.** LiteLLM и аналоги — это полезная абстракция, но они вводят тяжёлую зависимость в сервис, который в MVP делает один вызов одного эндпоинта. YAGNI: если завтра понадобится Anthropic-адаптер — добавим его как отдельный backend, а не переписываем весь слой.

**Конфигурация админом.** LLM-провайдер настраивается через environment variables Docker Compose:

```
FLINQ_LLM_BASE_URL=https://api.openai.com/v1
FLINQ_LLM_API_KEY=sk-...
FLINQ_LLM_MODEL=gpt-4o-mini
FLINQ_LLM_ENABLED=true
```

Флаг `FLINQ_LLM_ENABLED=false` — это реализация admin kill-switch из §10.3. При выключении AI reader должен продолжать работать: в карточках `new` слов пропадает AI-секция, остаётся только словарный перевод.

**Rate limiting и ретраи.** В MVP — простой retry с экспоненциальной задержкой (до 3 попыток) на сетевые ошибки и 5xx. Rate limiting в MVP отсутствует (вне scope, open question из §11).

**Журнал AI-вызовов.** В MVP пишется только в structured log. Отдельная таблица `ai_requests` с метаданными (input tokens, output tokens, latency, ошибки) — задел для Phase 2, когда появится admin dashboard. В MVP эта таблица может существовать как простой append-only аудит, если это не удорожает работу.

### Privacy, redaction и retention

Для self-hosted продукта с пользовательским контентом AI-интеграция не должна по умолчанию разносить сырые тексты по логам и аудитам. Поэтому фиксируются следующие правила:

**Что можно логировать в structured logs:**

- `request_id`;
- `user_id` или внутренний идентификатор пользователя;
- `lesson_id` и `language_pair`, если доступны;
- `provider`, `base_url`, `model`;
- latency;
- success/failure;
- HTTP status / provider error code;
- input token count и output token count;
- retry count;
- cache hit / miss.

**Что нельзя писать в structured logs по умолчанию:**

- сырой prompt целиком;
- сырой выделенный пользователем текст;
- полный текст урока;
- полный ответ модели;
- пользовательские заметки и переводы;
- API key и иные секреты.

**Append-only аудит `ai_requests`, если включён,** хранит только metadata-first запись:

- `request_id`;
- `user_id`;
- `lesson_id`;
- `provider`;
- `model`;
- `prompt_hash`;
- `selected_text_hash`;
- token usage;
- latency;
- success/failure;
- timestamps.

Сырые `prompt`, `selected_text` и `response_text` в таблицу `ai_requests` в MVP **не записываются**.

**Redaction policy.** Перед логированием и аудитом приложение обязано:

- удалять или маскировать `Authorization` и иные секреты;
- не сохранять body внешнего AI-запроса в обычные application logs;
- не сохранять body AI-ответа в обычные application logs;
- использовать hash для текстовых идентификаторов, когда нужен traceability без хранения сырого содержимого.

**Кэш.** Per-user, по ключу `(user_id, model, prompt_hash)`. При этом:

- в ключе не хранится сырой prompt;
- `prompt_hash` вычисляется от нормализованного prompt-а;
- кэш не является аудитом и не должен использоваться для аналитики;
- кэшированные записи не должны попадать в обычные application logs.

**Что хранится в кэше.** Для MVP допустимо хранить:

- `request_id`;
- `provider`;
- `model`;
- `prompt_hash`;
- нормализованный структурированный результат перевода;
- timestamp создания;
- timestamp последнего использования.

Хранение полного сырого ответа модели допускается только если это необходимо для повторного показа результата пользователю и если запись остаётся per-user. В таком случае эта запись не должна дублироваться в логах.

**Retention policy по умолчанию:**

- structured logs — по политике инсталляции, но приложение исходит из короткого retention, рекомендованно не более 14 дней;
- `ai_requests` аудит, если включён, рекомендованно не более 30 дней;
- AI-кэш — TTL 7 дней по умолчанию;
- администратор может ужесточить retention или полностью отключить аудит и кэш.

**Kill-switch и privacy-first режим.** При `FLINQ_LLM_ENABLED=false`:

- внешние AI-вызовы полностью отключены;
- новые записи в AI-кэш не создаются;
- `ai_requests` не пополняется новыми вызовами;
- reader продолжает работать только на словарных данных.

## Последствия

**Положительные:**

- Минимум кода: одна реализация, один набор тестов, один набор моков.
- Один и тот же код работает и с облачным OpenAI, и с локальным vLLM или LM Studio. Self-hosted аудитория получает реальную возможность работать off-line, не переписывая конфигурацию.
- Ollama имеет встроенный OpenAI-compat endpoint (`/v1/chat/completions`), поэтому покрывается бесплатно.
- Админ может поменять провайдера сменой env-переменных, без пересборки контейнеров.
- Зафиксирована privacy-граница: реализация не должна по умолчанию складывать пользовательский текст и ответы модели в логи.

**Отрицательные (принятые):**

- Anthropic Claude напрямую не поддерживается (у него свой формат сообщений и tool-calling). Пользователи, которые хотят Claude — могут ходить через OpenRouter как промежуточный слой. Если станет больно — добавим отдельный Anthropic-backend в Phase 2.
- Нет routing'а и fallback'а между несколькими провайдерами. Один адрес, одна модель. Расширение до multi-provider — не ломающее изменение, но нелокальное.
- Streaming ответов в MVP не используется (короткие перевод-ответы не выигрывают от стриминга). Если позже понадобится — OpenAI-compat API это поддерживает.
- Нет разницы между «system prompt» как админской настройкой и prompt template. В MVP prompt захардкожен в коде сервиса перевода, потому что сценарий один. Admin-editable prompt templates — Phase 2.
- Privacy-first логирование и короткий retention осложняют глубокую отладку prompt-level проблем. Для расследований придётся использовать временно включаемый debug-режим с отдельной явной настройкой, а не обычные production logs.

## Альтернативы (отвергнуты)

- **Отдельные адаптеры на каждого вендора.** Отвергнуто: YAGNI в MVP с одним сценарием использования.
- **LangChain / LiteLLM / Pydantic AI.** Отвергнуто: добавляет тяжёлые зависимости, скрывает поведение, усложняет отладку. Мы не используем 95% их возможностей.
- **Только локальный LLM через Ollama без поддержки облака.** Отвергнуто: противоречит целевой аудитории homelab + small team, где облачный GPT дешевле и удобнее для большинства.
- **Только облачный OpenAI без поддержки локальных.** Отвергнуто: противоречит self-hosted ценностям продукта и requirement'у спеки «возможность полностью отключить внешний AI».
