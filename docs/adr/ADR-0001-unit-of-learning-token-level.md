# ADR-0001 — Unit of learning: token-level, LingQ-style

- Статус: Accepted
- Дата: 2026-04-11
- Связан со: `docs/specs/2026-04-11-mvp-product-alignment-design.md` §4

## Контекст

Спецификация продукта §16.2 формулирует один из самых ранних и самых дорогих для пересмотра выборов — что именно является «единицей обучения» в Flinq. Варианты, рассмотренные в §16.2:

- **A. Token + Lemma + Phrase** как три отдельные сущности. Token ссылается на Lemma; пользователь может сохранять любой уровень; статусы и SRS идут по Lemma (для слов) и по Phrase (для фраз); статистика чтения — по Token. Это рекомендация спеки.
- **B. Только Lemma + Phrase**, без отдельной сущности Token.
- **C. Token + Phrase, без Lemma** — каждая словоформа отдельная учебная единица; ровно так, как устроен LingQ.

Ключевой аргумент за Lemma — корректная работа на языках с богатой морфологией (RU): если пользователь видит `книга` и `книги`, интуитивно это одно слово. Без лемматизации пользователь будет встречать одно слово по несколько раз в разных формах в SRS-сессии.

Ключевой аргумент против Lemma — зависимость от морфологических анализаторов (`pymorphy3`, `spaCy`, `stanza`), сложность разрешения омонимии, дополнительная таблица и индексы в БД, непривычный для LingQ-юзеров UX.

## Решение

Принят **вариант C**: учебная единица = Token (поверхностная словоформа). Phrase — отдельная first-class сущность. `Lemma` **не является** сущностью в схеме данных.

**Нормализация Token при сохранении:**

1. Unicode normalization → NFC.
2. `lowercase`.
3. Обрезание ведущей и замыкающей пунктуации (`.,;:!?"'«»()[]—…`).
4. Диакритика **сохраняется**: `cafe` ≠ `café`, `mao` ≠ `mão`.
5. Внутренние дефисы и апострофы **сохраняются**: `well-known` и `don't` — валидные токены.

**Статистика и статусы.** Все счётчики (прочитано, learning, known) считаются на уровне Token. Каждая словоформа имеет собственный статус (`new` / `learning` / `known`), собственный перевод, собственную SRS-историю.

**Phrase** хранится как отдельная сущность со своим text-полем (нормализованным по тем же правилам), собственным статусом и собственным SRS-item. Матчинг phrase в тексте — по точному совпадению последовательности Token'ов (open question из §11 — распространение матчинга на другие уроки).

**SRS очередь — единая** для Token и Phrase. Формат review-карточки может различаться (плоская «слово ↔ перевод» для Token, cloze для Phrase), но расписание, алгоритм, дневной лимит и история — общие.

**Создание записи и редактирование формы.** При создании `Token`-записи из reader сохраняется **ровно та словоформа, которая была в тексте** после нормализации, описанной выше. Пользователь не может изменить `books` на `book` или `книги` на `книга` в карточке добавления.

**После создания записи** текст `Token` считается identity-defining полем и также не редактируется. Нельзя задним числом изменить уже созданную запись `books` на `book`.

Если пользователю нужна другая форма слова, это делается не через редактирование существующего Token, а через **отдельный интерфейс управления словарём**, где можно вручную создать новую словарную запись, не привязанную к конкретному вхождению в тексте.

Перевод, заметки и другие пользовательские поля при этом остаются редактируемыми после создания.

## Последствия

**Положительные:**

- Схема данных упрощена: две основные таблицы (`tokens`, `phrases`) вместо трёх, без Token↔Lemma связей и без таблицы словоформ.
- Нет зависимости от морфологических библиотек в ядре — ядро работает с любым языком, где определён алгоритм токенизации и понятие «слова».
- UX повторяет LingQ и знаком части целевой аудитории.
- Нет проблемы омонимии (`стали` как форма «стать» и как сущ. «сталь» — в LingQ-модели это просто одна запись с ручным переводом).
- Контекстный AI-перевод становится ещё более ценным, потому что компенсирует отсутствие леммы: AI в контексте всегда поймёт, это глагол или существительное.
- После создания identity записи стабильна: статус, SRS-история и статистика не переписываются задним числом из-за редактирования текста.
- Reader-flow остаётся строго token-level и ближе к оригинальной LingQ-модели: пользователь сохраняет именно ту форму, которую увидел в контенте.

**Отрицательные (принятые):**

- В SRS пользователь действительно встретит `poder`, `pode`, `pudesse`, `pudessem` как отдельные карточки. Это может удлинить review-очередь для морфологически богатых языков (RU, PT).
- Счётчик «известных слов» растёт быстрее, чем это было бы на lemma-level, и не отражает уникальный словарный запас в лингвистическом смысле. Нужно честно коммуницировать в UI, что это именно «словоформы», а не «лемм».
- Сложнее добавлять грамматические фичи (сравнение форм, склонение по таблице) — все такие подсказки придётся делать через AI, а не через структурированную словарную базу.
- Ранний отказ от Lemma закрывает некоторые будущие фичи (phrase mining на lemma-level, lemma-based coverage estimate). Если захотим их — надо будет делать дополнительный слой поверх Token.
- Пользователь не может нормализовать словоформу прямо в reader-flow. Если ему нужна отдельная словарная форма (`book` вместо `books`), её надо добавлять отдельной записью через vocabulary management UI.

**Обратимость.** Решение обратимо ценой миграции: добавление Lemma-таблицы и backfill-процесса для существующих Token. Но само изменение продуктового поведения (один Token → группа форм одной леммы) будет заметным для пользователя, что делает обратимость скорее «теоретической».

## Альтернативы (отвергнуты)

- **Token + Lemma + Phrase** (рекомендация спеки). Отвергнуто: сложность ядра, зависимости от морфологических библиотек, непривычный для LingQ-аудитории UX.
- **Только Lemma + Phrase**. Отвергнуто: теряется возможность тонкой работы с конкретной словоформой, не решает проблему морфологической зависимости.
