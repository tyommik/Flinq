# ADR-0005 — Word status model: LingQ-style with confidence levels

- Статус: Accepted
- Дата: 2026-04-11
- Supersedes: `docs/adr/ADR-0002-word-status-model-and-reader-ui.md`
- Связан со: `docs/specs/2026-04-11-mvp-product-alignment-design.md` §9, ADR-0001, `docs/architecture/2026-04-11-mvp-architecture-overview.md` §7.4, §10.3

## Контекст

ADR-0002 ввёл упрощённую трёхстатусную модель `new / learning / known` с per-page bulk-known переходом. На этапе архитектурного проектирования (см. architecture overview §7.4, §10.3) выяснилось, что эта модель недостаточно выразительна для LingQ-style флоу, на который ориентируется Flinq:

1. **Отсутствие `ignored`.** В LingQ пользователь может явно пометить слово как «Ignore» — типичный кейс для имён собственных, чисел, мусорных токенов (плохо сегментированные), редких технических терминов. Это семантически не «known», а «не считай это словом для целей обучения». В трёхстатусной модели ADR-0002 такие слова попадают в `known`, засоряют метрику «известных слов» и становятся неотличимы от реально знакомой лексики.

2. **Отсутствие градаций внутри «учусь».** В LingQ есть четыре уровня уверенности (LingQ Status 1..4) + отдельный Known. Они обновляются постепенно по мере успешных повторений, давая SRS-алгоритму пространство для плавного продвижения. В модели ADR-0002 нет confidence — SRS либо держит item в `learning`, либо one-shot'ом промотирует в `known`.

ADR-0002 явно рассмотрел оба аргумента в разделе «Альтернативы (отвергнуты)» и отверг их ради простоты MVP. Мы пересматриваем это решение: упрощение оказалось слишком сильным, и восстановление LingQ-флоу сейчас дешевле, чем доделывать его позже поверх написанного кода и уже накопленных пользовательских данных.

## Решение

### Статусы

Четыре значения:

- **`new`** — токен встречен в тексте, но никаких действий с ним пользователь не производил.
- **`tracked`** — токен явно сохранён пользователем для изучения. Имеет `confidence` в диапазоне `0..5`. Участвует в SRS.
- **`known`** — токен явно помечен как знакомый, не требует внимания, не участвует в SRS.
- **`ignored`** — токен явно исключён из обучения. Не участвует в SRS. **Не входит в счётчик `known`.**

Модель применяется и к `Token`, и к `Phrase` (см. ADR-0001) — эти сущности имеют собственные независимые значения статуса.

### Confidence внутри `tracked`

Значения `confidence`: целое число в диапазоне `0..5` (шесть уровней).

- `0` — только что добавлено, пользователь ещё не проходил SRS по этому item'у.
- `1..4` — промежуточные уровни. Обновляются SRS-алгоритмом по результатам ответов пользователя (правильный ответ повышает, ошибка понижает).
- `5` — максимальная уверенность внутри `tracked`. Следующее успешное повторение (либо ручное действие) промотирует item в `known`.

Точные правила продвижения и понижения confidence зависят от выбранного SRS-алгоритма и остаются открытым вопросом (см. «Открытые вопросы» ниже и `decision log §11`).

Пользователь может вручную менять confidence через карточку в reader'е — например, «я и так уверен в этом слове, поставь 4» или «сбрось confidence, давай учить заново».

### Переходы между статусами

| Из | В | Триггер |
|---|---|---|
| `new` | `tracked` | клик по слову + «Add to study» в карточке; `confidence` стартует с `1` (или с выбранного пользователем значения при сохранении) |
| `new` | `known` | **per-page bulk:** при нажатии «next page» все `new` occurrences на текущей странице становятся `known` |
| `new` | `ignored` | клик по слову + «Ignore» в карточке |
| `tracked` | `tracked` (`confidence ±1`) | ответ в review session (правильно → `+1`, ошибка → `-1`, не опускаясь ниже `0`) |
| `tracked` | `known` | SRS-graduation по достижении `confidence = 5` и очередного успешного повторения, **или** ручная пометка «Move to known» в карточке |
| `tracked` | `ignored` | ручная пометка «Move to ignored» в карточке |
| `known` | `tracked` | ручной revert «Move back to tracked» в карточке (edge case) |
| `ignored` | `tracked` | ручной revert «Reactivate» в карточке (edge case) |
| `known` ↔ `new`, `ignored` ↔ `new`, `ignored` ↔ `known` | — | **не поддерживаются напрямую**; при необходимости идти через `tracked` |

**Per-page bulk-known** действует **только на `new` occurrences**. Токены в `tracked`, `known`, `ignored` при переходе на следующую страницу не затрагиваются. Reader обязан поддерживать **undo последнего bulk-known действия**.

### Визуальное представление в reader

| Статус | Отображение в тексте |
|---|---|
| `new` | бледно-голубой фон |
| `tracked` | жёлтый фон; опционально тон/насыщенность варьируется в зависимости от `confidence` (низкая — более насыщенный жёлтый, высокая — светлее). В MVP допустим равномерный жёлтый без градации |
| `known` | без выделения |
| `ignored` | без выделения; в MVP визуально неотличим от `known` |

Детали confidence-gradient — предмет имплементации reader-UI, не архитектурное решение. Визуальный индикатор `ignored` (например, тонкое подчёркивание dotted) — open question на MVP.

### Поведение при клике на слово

| Статус | Содержимое карточки |
|---|---|
| `new` | AI-перевод первым (с меткой AI-generated) → словарный вариант → кнопки: **«Add to study»** (→ `tracked`, `confidence 1`), **«Ignore»** (→ `ignored`), **«I know this»** (→ `known`) |
| `tracked` | Сохранённый пользовательский перевод как первичный → индикатор `confidence 0..5` → кнопки: **«Edit translation»**, **«Adjust confidence»**, **«Move to known»**, **«Move to ignored»** |
| `known` | Комбинированный AI + словарный перевод (пользователь всё ещё может захотеть подсказку) → кнопка **«Move back to tracked»** (edge case) |
| `ignored` | Короткая заметка «Ignored» + оригинал токена → кнопка **«Reactivate»** (→ `tracked`) |

### Поведение при клике на фразу

Выделение нескольких соседних токенов создаёт или находит существующую `Phrase`-запись. Далее применяется та же статусная модель (`new / tracked / known / ignored`) и тот же confidence-диапазон. Визуально фразы используют тот же цветовой код.

### Метрики

Метрики MVP пересмотрены:

- **Прочитанные токены** — без изменений.
- **Счётчик `tracked`** (слова + фразы) — заменяет прежний счётчик `learning`.
- **Счётчик `known`** (слова + фразы) — без изменений по смыслу, но теперь **не включает `ignored`**.
- **Счётчик `ignored`** (слова + фразы) — новая вспомогательная метрика. Хранится, но в MVP **не показывается** как hero metric. Доступна в стат-экране для пользователей, которым это важно.

Именно отделение `ignored` от `known` — главное продуктовое улучшение по сравнению с ADR-0002.

## Последствия

**Положительные:**

- Метрика `known` становится чище: имена собственные, числа, мусорные токены больше её не загрязняют.
- SRS получает пространство в виде 6 уровней confidence для плавного продвижения item'а к `known`. Это убирает ситуацию, когда item резко перескакивает из `learning` в `known`.
- Пользователь получает явный способ пометить токен как «не учить» без побочных эффектов в статистике.
- Флоу ближе к реальному LingQ — предсказуемее для пользователей, знакомых с LingQ.
- Карточки со статусом становятся продуктивнее: можно выразить «я и так это знаю, не надо тренировать», «я пропускаю это слово», «я учу это с такой-то уверенностью» — каждое через явное действие.

**Отрицательные (принятые):**

- Схема данных `token_items` и `phrase_items` усложняется: enum статусов становится `new | tracked | known | ignored`, появляется поле `confidence SMALLINT` (`0..5`, NULL для не-tracked).
- Reader UI сложнее: четыре статуса вместо трёх, больше кнопок в карточке, обработка опционального confidence-gradient.
- Переходы требуют явных тестов state machine — особенно manual reverts из `known` и `ignored`.
- SRS-алгоритм должен оперировать 6-уровневой шкалой confidence, а не просто bucketing «learning/graduated». Это делает выбор алгоритма (SM-2 vs FSRS vs custom) более чувствительным.
- Miграция метрик: `ignored` теперь не входит в `known`. Если у кого-то уже есть накопленные данные по ADR-0002 — им потребуется конвертация (на момент принятия ADR кода ещё нет, так что migration path пустой).

**Обратимость.** Откат к трёхстатусной модели ADR-0002 возможен через миграцию схлопом `ignored` в `known` и обнулением confidence, но это потеря пользовательских данных (разделение known/ignored) и потеря тонкости прогресса (градации внутри tracked). Практически решение зафиксировано.

## Открытые вопросы

- **SRS graduation rule.** Сколько успешных повторений при `confidence = 5` нужно для автоматического перехода в `known`? Зависит от выбора SRS-алгоритма. Добавляется к open question «точный алгоритм SRS» в decision log §11.
- **Confidence visual gradient.** Нужно ли в MVP визуально отличать `confidence 1` от `confidence 4` в тексте, и если да — как. Допустимо в MVP использовать равномерный жёлтый.
- **`ignored` visual.** В MVP `ignored` неотличим от `known`. Если появится UX-запрос — добавить тонкий hint (dotted underline, точка в углу) как post-MVP улучшение.
- **Bulk-ignored.** Массовая операция «пометь все не-алфавитные токены (числа, знаки) как ignored» — логичное расширение, но не в MVP.
- **Bulk-known undo.** Reader должен поддерживать undo последнего bulk-known действия. Это инвариант, а не open question — упомянуто здесь для трассируемости.

## Альтернативы (отвергнуты)

- **Сохранить 3 статуса из ADR-0002 (`new / learning / known`).** Отвергнуто: не отделяет ignored от known, засоряет метрику «известных слов», не даёт SRS пространства для плавного продвижения.
- **5+ статусов, где каждый уровень уверенности — отдельный статус** (как в некоторых других reader-приложениях). Отвергнуто: излишняя сложность state machine и UI, LingQ-стиль `0..5` внутри одного статуса `tracked` достаточен и привычен.
- **Confidence `0..3` или `0..10`.** Отвергнуто: `0..5` — LingQ-конвенция (LingQ использует Status 1..4 + Known, что даёт 5 шагов; мы расширяем до 6, включая `0` как «только добавлено»). Даёт и простор для SRS, и не перегружает UI.
- **Хранить confidence отдельной таблицей (`srs_state`), не в `token_items`.** Отвергнуто: confidence жёстко связан со статусом и item'ом, отдельная таблица лишь добавит JOIN без выигрыша.
