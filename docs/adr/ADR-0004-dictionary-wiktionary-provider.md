# ADR-0004 — Dictionary: Wiktionary dump + provider interface

- Статус: Accepted
- Дата: 2026-04-11
- Связан со: `docs/specs/2026-04-11-mvp-product-alignment-design.md` §8, §10.2

## Контекст

Спецификация §7.5 требует наличия встроенного двуязычного словаря под каждый изучаемый язык, с поддержкой нескольких источников, частей речи, примеров и пользовательских заметок. §16.6 формулирует позицию «гибрид: словарь первичен, AI как дополнение».

Для MVP требуется:

1. Рабочий двуязычный словарь для пар EN↔RU, EN↔PT, RU↔PT, доступный **без внешних вызовов** (см. kill-switch AI в §10.3).
2. Интерфейс, позволяющий в будущем добавлять дополнительные источники словарных данных (например, Lingvo, собственный сборщик, коммерческие API) без переработки слоя перевода.
3. Приемлемая лицензионная модель для self-hosted распространения.

Основные кандидаты для данных:

- **Wiktionary** — open dump через Kaikki.org или Wikidata, CC-BY-SA. Покрытие EN/RU/PT хорошее, структурированные данные (часть речи, примеры, переводы, формы), но качество неоднородное.
- **Коммерческие словари** (Oxford, Collins, Lingvo). Качественные, но проприетарные, не распространяются с self-hosted продуктом.
- **FreeDict / StarDict** — более старые dump'ы, часто xdxf или dictd формат, покрытие уступает Wiktionary для редких пар.
- **LLM-генерация словарных статей** поверх AI. Работает, но это не «словарь» — это AI, и противоречит decision'у «базовый режим без AI».

## Решение

### Источник данных

**Wiktionary dump** через [Kaikki.org](https://kaikki.org/) как первичный источник словарных данных для EN, RU, PT. Лицензия CC-BY-SA 4.0 (то же, что и у самой Wiktionary).

Kaikki предоставляет дампы в JSONL-формате со структурированными данными (lemma, часть речи, значения, переводы, примеры, этимология). Это более пригодно для программной обработки, чем сырой XML-дамп Wiktionary.

**Форма хранения в Flinq:** при инициализации (setup-скрипт при первом запуске Docker Compose) дамп загружается, парсится и загружается в таблицы `dictionary_entries` / `dictionary_translations` / `dictionary_examples`. Исходные JSONL-файлы не хранятся, но версия дампа (дата, источник) фиксируется в метаданных.

Обновление словарных данных — ручное действие админа (`make dictionary-refresh` или аналог).

### Provider interface

В слое перевода определяется интерфейс `DictionaryProvider` со следующими методами:

```
lookup(token: str, from_lang: str, to_lang: str) -> DictionaryLookupResult
lookup_phrase(text: str, from_lang: str, to_lang: str) -> DictionaryLookupResult
```

Реализации:

- `WiktionaryDictionaryProvider` — чтение из внутренней БД Flinq, заполненной из Kaikki-дампа. Единственная реализация в MVP.
- Будущие провайдеры (commercial API, дополнительные open-source словари) реализуют тот же интерфейс и регистрируются в админских настройках.

**Приоритет источников.** В MVP один источник, поэтому вопрос приоритетов не стоит. Как только появится второй провайдер, нужна явная политика (выбор админом или через chain-of-responsibility). Это — post-MVP работа.

### Атрибуция лицензии

CC-BY-SA требует атрибуции. В UI Flinq на карточке слова, когда показаны словарные данные, отображается источник и ссылка (например, «From Wiktionary (CC-BY-SA)»). Это не факультативно — это требование лицензии и должно быть реализовано как жёсткая часть карточки, а не как admin-отключаемая опция.

Раздел «Legal notices» в UI (About/Credits) перечисляет все используемые источники данных с их лицензиями.

### Пользовательский перевод

Пользовательский перевод, сохранённый в `Token` или `Phrase` записи, имеет безусловный приоритет при отображении в карточке для статусов `learning`. Для `new` и `known` он не применяется (там пользовательского перевода ещё/уже нет).

Пользовательский перевод **не экспортируется** обратно в словарную базу — он живёт только в личном словаре. Это разделение важно: база Wiktionary неизменна между пользователями одной инсталляции, персональные переводы — per-user.

## Последствия

**Положительные:**

- Продукт работает без внешних вызовов (требование §8.5 спеки и §10.3 decision doc'а).
- Лицензионно чист для self-hosted распространения при условии корректной атрибуции.
- Покрытие EN/RU/PT и основных пар — достаточное.
- Provider interface оставляет возможность добавлять дополнительные источники без переработки слоя.
- Процесс обновления данных явный и воспроизводимый: один дамп, одна дата, одна команда.

**Отрицательные (принятые):**

- Начальный setup инстанса требует загрузки нескольких сотен мегабайт данных. Это заметный размер для homelab и требует документации.
- Качество словарных статей Wiktionary неоднородно. Для некоторых редких слов переводы будут неполными или отсутствовать. Предполагается, что в таких случаях пользователь идёт через AI (если AI включён) или вводит свой перевод.
- CC-BY-SA обязывает показывать атрибуцию даже в пользовательском UI. Это не проблема, но забыть — легко.
- Нет встроенной механики периодических обновлений. Если Wiktionary поменяется радикально — нужна ручная пересборка. Для MVP это приемлемо.

**Обратимость.** Замена источника данных — локальное изменение: меняется ETL-скрипт и, возможно, схема таблиц словаря. Ядро и provider interface не меняются. Сохранённые пользовательские переводы не зависят от источника и не затрагиваются.

## Альтернативы (отвергнуты)

- **Коммерческие словарные API как обязательный источник.** Отвергнуто: противоречит self-hosted ценностям и бесплатной доставке продукта.
- **LLM-only как замена словарю.** Отвергнуто: противоречит requirement'у «продукт работает без внешних AI-вызовов».
- **FreeDict/StarDict dumps.** Отвергнуто: хуже покрытие, более сложные legacy-форматы, менее структурированные данные.
- **Свой скрапер Wiktionary XML.** Отвергнуто: Kaikki.org уже делает эту работу надёжно, дублировать бессмысленно.
- **Chain-of-providers с автоматическим fallback в MVP.** Отвергнуто: YAGNI, единственный провайдер в MVP.
